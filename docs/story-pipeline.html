1. Overview & modes

Three modes have existed historically; only unified is supported for new features. Legacy pictureBook and outlineAndText modes remain only to read older stories.

2. Trigger & job lifecycle

  • Frontend wizard (client/src/pages/StoryWizard.tsx) calls POST /api/jobs/create-story with characters, settings, art style, language, optional location.
  • Server creates a story_jobs row, returns jobId immediately, and runs the pipeline in the background.
  • Client polls GET /api/jobs/:id/status; the response carries the partial story (sceneImages start null, fill in as stages complete).
  • The job records timestamps per stage and a partial JSON checkpoint so the UI can render incrementally and the server can resume on crash.
  • Cancellation: POST /api/jobs/:id/cancel trips an AbortController propagated through every async stage.
A push to master auto-deploys to Railway, kills the Node process, and aborts every in-flight job. Real users mid-story lose their work. See operations §deploy for the mandatory approval flow.

3. Unified Claude pass

One messages.create call returns the entire story payload — outline, visual bible (characters + objects + locations), per-page text, and per-scene hints (shot, framing, primary characters, optional location reference). Prompt: prompts/unified-pipeline.txt (and language-specific variants).

Why one call: keeping outline + text + scene hints in the same context guarantees consistency. A page's text references the same hat the outline introduced; a scene hint about "the marketplace at dusk" aligns with the outline's Act II setup.

What the unified pass produces

  • title, titleCandidates (alternatives extracted by parser.extractTitleCandidates).
  • outline with act structure.
  • visualBible — characters, objects, locations, with id, name, traits, dbKey for DB-linked entities.
  • scenes[] — one per page; includes text, shot, framing, primaryCharacters, hint.objects (location IDs), hint.events.
  • language, artStyleId, readingLevel.

4. Scene expansion (Art Director)

For each scene, a second Claude call (scene-expansion.txt) produces the illustration brief used downstream:

  • sceneDescription — Art Director prose with character interactions, props, mood.
  • textPosition — one of 6 positions; spread rule: odd pages = left, even pages = right. Corrected by enforceSpreadTextPosition if Claude picks the wrong side. See text overlay.
  • emptyScenePrompt — background-only prompt used if iterative placement is enabled.
  • interactions — declared character interactions; the semantic eval scores against these.

Locations: when a scene's hint.objects contains a LOC### ID matching a curated historical_locations row, the prompt is augmented with the location name + dbKey + context. See landmarks.

5. Image generation

Generation lives in server/lib/images.js. The image prompt is built by buildImagePrompt() in server/lib/storyHelpers.js and combines:

  • Art style block (loaded from client/src/constants/artStyles.ts).
  • Scene description (Art Director output).
  • Character lines (one per primary character, with shot-aware reference packing — see character system §shot-aware).
  • COPY SPACE instruction for the calm text zone.
  • No-character-name rendering rule (no labels, signs, captions reading the prompt's character names).

Reference modes (per character): strict (full grid), loose (face + body crops), styled-only, off. Default is loose. Wide shots receive only the body crop; non-wide shots receive face + body as separate refs.

Iterative placement (optional)

generateWithIterativePlacement() generates the empty scene first, then composites characters via Grok edit. Improves multi-character placement when the scene metadata declares background-depth characters. Disabled by default (generateEmptyScenes: false in server/config/models.js).

6. Evaluation & repair

Three evaluators run in parallel after image generation:

  1. Quality eval (Gemini 2.5 Flash) — qualityScore 0-100 + fixableIssues[]. Prompt: image-evaluation.txt.
  2. Semantic eval — does the image match the scene description? Penalties baked into finalScore. Prompt: image-semantic.txt.
  3. Entity consistency — bbox detection + cascade face merge + character-presence check. finalChecksReport.entity.

The repair workflow is a separate piece — full details in image quality stack. Summary: pages scoring below threshold or with too many fixable issues are redone; up to 3 global passes; character repair runs before pick-best so all variants compete.

7. Covers

Three cover types per story:

  • frontCover — title image with a few characters.
  • initialPage — dedication / inside-front image.
  • backCover — back image with story summary blurb.

Generated by server/lib/coverIterate.js. The cover scene metadata always merges LOC### IDs from hint.objects so curated landmark photos reach the cover. See covers & composite for the full cover stack and the alternative scene-composite path.

8. Persistence & output

The final story is written to:

  • stories.data JSONB — full payload (sceneImages with imageData/imageUrl, textPosition, textRect, sceneDescription, retryHistory, imageVersions; coverImages; characters snapshot).
  • story_images table — versioned rows per page (active + history) with R2 URLs.
  • story_jobs.partial updates throughout for resumability.

PDF rendering (server/lib/pdf.js) reads from this payload on demand for download / print. Print path uses Gelato (server/routes/print.js).
