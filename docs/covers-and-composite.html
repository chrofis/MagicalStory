Covers & Scene Composite

Cover generation (front, initial page, back) and the optional scene-composite pipeline that splits a busy scene into multiple Grok calls and stitches the result.

1. Cover types

Each story has three cover items, all stored under stories.data.coverImages:

TypePurposeSource prompt
frontCoverTitle image — main characters in a hero composition.prompts/front-cover.txt
initialPageInside-front / dedication page.prompts/initial-page-*.txt (per language)
backCoverBack cover with story summary blurb space.prompts/back-cover.txt

Each cover type has its own imageVersions[] with the same pick-best logic as story pages — see image quality §pick-best.

2. Cover generation flow

  1. Cover scene metadata — assembled from the story outline. The cover scene always merges LOC### IDs from hint.objects so curated landmark photos reach the cover (without this, covers regularly missed the iconic location).
  2. Reference packing — same shot-aware logic as page images (character system §shot-aware). Cover usually wants 2-3 hero characters at medium shot.
  3. Title text injection — coverText option passes the story title into the prompt so the model leaves room for it. The PDF renderer overlays the actual text.
  4. Quality gate — same evaluators as pages.
  5. Iteration — UI exposes "Iterate cover" which calls POST /:id/covers/iterate with feedback.

Implementation: server/lib/coverIterate.js. Frontend: client/src/components/generation/CoverPanel.tsx.

3. Multi-character limits

Empirical limit: 3-4 hero characters per cover with the standard pipeline. Tests with 6-10 characters hit recurring issues — face mismatch, character drop-out, cluttered composition. See archive (MULTI-CHARACTER-COVER-TESTS) for the full test log.

For families / classrooms / large casts, the scene-composite pipeline (next section) is the alternative path.

4. Scene composite pipeline

Behind MODEL_DEFAULTS.enableSceneComposite (on in staging). Production code: server/lib/sceneComposite.js. The pipeline runs per page, building each illustration from a clean empty plate plus per-character cutouts pasted at scene-prescribed positions and blended together.

Five steps

  1. Populated plate — Grok generate (text-to-image) paints the full scene + all coloured silhouettes in one call. The setting, VB props, and silhouettes are committed together so the downstream depopulate step has a single self-consistent source. This replaces the older "empty BG first, then Grok-edit silhouettes onto it" pair — the old blocking edit silently drifted the background (added a bench, swapped a VB prop, repainted the floor) so the empty plate stopped matching the silhouette plate.
  2. Detect bboxes — colour-blob detection (findColorBbox) finds each silhouette in the populated plate, merges any vertically-stacked fragments (table or fence cutting a figure in half).
  3. Depopulate (derive clean BG) — Grok edit on the populated plate with a "remove every flat-colour silhouette, repaint the regions with surrounding terrain, keep every other pixel identical" prompt. Produces a clean BG that is guaranteed to match the world geometry + VB props of the populated plate exactly. This is the plate the cutouts get pasted onto.
  4. Composite cutouts — the character's cutout is either:
    • Static cell (default): one of 8 cells is cropped from the character's pre-generated 2×4 reference sheet, background-removed, optionally flipped, scaled to the detected bbox.
    • Phantom-pose render (MODEL_DEFAULTS.phantomPoseRender: true): the cropped silhouette region + the 2×4 sheet are sent to Grok edit, which renders that character in the silhouette's exact pose (sitting, climbing, reaching) on a white background. One Grok call per character per page.
    Cutouts are sharp-composited onto the derived clean BG at each detected bbox, z-ordered by foot-Y so closer characters paint on top.
  5. Blend — Grok edit pass on the composited scene. Image 1 is the canvas; Image 2 is the Visual Bible portrait grid (labelled face/costume reference). The prompt is a tight DO/DON'T list: blend lighting, soften pasted edges, add ground shadows, paint over silhouette-residue colour fringes; do NOT move/resize/restage characters; DO add named REQUIRED OBJECTS from the brief that the staged composite is missing (treasure chest, parchment map, etc.).

Cast sizeStatic-cell variantPhantom-pose-render variant

4 characters3 calls / $0.063 + 4 = 7 calls / $0.14
6 characters3 calls / $0.063 + 6 = 9 calls / $0.18

Plus a one-time per-story sheet cost: each character × each costume × 1 Grok call ($0.02 each) for the 2×4 reference sheet. Cached on the character record, reused across pages.

Pros

  • Character identity is very close to the source photo. The 2×4 sheet locks face + costume; phantom-pose-render then keeps that identity while reposing. End result: characters look like the real person, not a generic illustration of "a child".
  • Position, depth, size are dictated, not suggested. The silhouette in step 2 sets the bbox; step 3 pastes the cutout at that bbox. Foreground/midground/background separation is reliable in a way prose ("Noah in left foreground") never is for a single-pass model.
  • Per-character action is enforced by silhouette shape. If the silhouette shows arms reaching, the cutout fits that shape. The model can't quietly default to "two characters standing facing camera".
  • Splittable figures. When an occluder (table, parapet) cuts a character into two coloured fragments, the bbox detector merges them — both halves of the character get composited correctly.
  • Required objects get added by the blend step. Even when the empty scene omitted a treasure chest or the cutout doesn't carry a prop, the blend prompt names the object and Grok paints it in.

Cons

  • Expensive on cast-heavy pages. 6 characters with phantom-pose-render = 9 Grok calls per page (~$0.18), versus 1 call ($0.02) for a single direct-path generation. A 14-page story with 5 main characters runs ~$2-3 in image generation alone.
  • The image still often doesn't look fully natural. The blend step has to harmonise lighting and soften pasted edges; in practice you can sometimes still see slightly mismatched lighting, stiff posing, or compositional awkwardness that a single-pass model would have avoided by drawing the scene holistically.
  • Many failure modes. Each of the four steps can fail: Grok desaturates the silhouette (bbox detector misses it), the 2×4 sheet has a small figure (cutout upscales to blur), the blend step paints over a needed prop, the phantom-pose Grok call drifts off-style. Quality gates and retries catch most but not all.
  • Style adherence is weaker than a single-pass model. Three to four sequential Grok edits accumulate small style drift. Watercolor washes thin out; line work hardens. The blend step's "render in this style" instruction helps but can't fully recover.
  • Latency. A 4-page composite story takes ~7-8 minutes wall-clock vs ~3-4 min for a direct-path story of the same size.

When to prefer composite over direct

The cast-aware dispatcher is live in server/lib/imageRouter.js: cast 0 → direct/refMode=off; cast 1–3 → direct/refMode=loose; cast 4–5 → composite; cast 6+ → composite + phantom-pose. inputData.composite / inputData.phantomPoseRender per-story overrides hard-pin the choice. See image-generation-methods §routing.

5. The 2×4 character sheet

Per character per costume, generated once and cached on the character record. server/lib/character2x4Sheet.js sends three reference images to Grok edit:

  1. Phantom asset (server/assets/phantom-watercolor-{tier}-axes.png; tier picked by age via loadPhantom) — a 2×4 grid of wooden mannequins showing the 8 target angles, with a 3-axis RGB gizmo (red X / green Y / blue Z) overlaid on the face region of each cell. The gizmo communicates head orientation per cell without giving Grok any facial geometry to copy — earlier versions baked in eye-dots and a mouth-line, both of which leaked into the rendered character ("phantom face leak"). The model is told to ignore the silhouette and surface texture; the gizmo is unmistakably non-anatomical so it gets dropped too. See docs/decisions.md → "Phantom face replaced with RGB axis-gizmo overlay".
  2. Standard avatar — the character's body / proportions reference.
  3. Face photo — the character's face / hair / skin tone reference.

Output is a 2×4 sheet: cells 1-4 are head-and-neck views (front, 3⁄4, profile, back-of-head); cells 5-8 are full-body in costume at the same four angles. A Gemini-based quality gate evaluates face + costume match scores and retries (up to MAX_STYLED_AVATAR_RETRIES) if either falls below threshold.
