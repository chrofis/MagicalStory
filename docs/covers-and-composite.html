Covers & Scene Composite

Cover generation (front, initial page, back) and the optional scene-composite pipeline that splits a busy scene into multiple Grok calls and stitches the result.

1. Cover types

Each story has three cover items, all stored under stories.data.coverImages:

TypePurposeSource prompt
frontCoverTitle image — main characters in a hero composition.prompts/front-cover.txt
initialPageInside-front / dedication page.prompts/initial-page-*.txt (per language)
backCoverBack cover with story summary blurb space.prompts/back-cover.txt

Each cover type has its own imageVersions[] with the same pick-best logic as story pages — see image quality §pick-best.

2. Cover generation flow

  1. Cover scene metadata — assembled from the story outline. The cover scene always merges LOC### IDs from hint.objects so curated landmark photos reach the cover (without this, covers regularly missed the iconic location).
  2. Reference packing — same shot-aware logic as page images (character system §shot-aware). Cover usually wants 2-3 hero characters at medium shot.
  3. Title text injection — coverText option passes the story title into the prompt so the model leaves room for it. The PDF renderer overlays the actual text.
  4. Quality gate — same evaluators as pages.
  5. Iteration — UI exposes "Iterate cover" which calls POST /:id/covers/iterate with feedback.

Implementation: server/lib/coverIterate.js. Frontend: client/src/components/generation/CoverPanel.tsx.

3. Multi-character limits

Empirical limit: 3-4 hero characters per cover with the standard pipeline. Tests with 6-10 characters hit recurring issues — face mismatch, character drop-out, cluttered composition. See archive (MULTI-CHARACTER-COVER-TESTS) for the full test log.

For families / classrooms / large casts, the scene-composite pipeline (next section) is the alternative path.

4. Scene composite pipeline

Behind a feature flag (enableSceneComposite). Splits a busy scene into three Grok calls:

  1. Phantom call — generate the empty scene background (no characters).
  2. Cell calls — generate each character or character-group as an isolated cell against a neutral background.
  3. Stitch call — composite all cells onto the phantom and pass through Grok edit for blending.

Output: a 2×4 sheet showing each intermediate stage (phantom + cells + final). Useful for debugging large casts and for diagnosing which character broke. Implementation: scripts/test-scene-composite-smoke.js for smoke tests; production code lives in server/lib/sceneComposite.js (gated by flag).

Why three calls instead of one

Single-prompt multi-character generation degrades roughly with character count (face quality, prompt adherence, costume drift). Isolating each character preserves identity at the cost of more API calls and a stitch step that introduces seams. The composite pipeline accepts that trade for high-cast scenes only.

5. Composite cell map

The cell map (Visual Bible-aware) declares per cell:

  • Character ID(s) inside the cell.
  • Target placement on the phantom (xy + scale).
  • Costume / pose overrides if the scene calls for non-default clothing.
  • Outline schema reference (which scene event the cell renders).

The phantom has a coordinate system; cells declare their target box; the stitch step composites with feathering. Costume overrides flow through resolveStyled2x2 which reads the canonical styledAvatars path.
