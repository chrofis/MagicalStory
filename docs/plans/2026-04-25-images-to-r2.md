# Migrate image bytes from Postgres to Cloudflare R2

**Status:** plan only — not started
**Owner:** roger
**Created:** 2026-04-25

## Why

The Postgres DB is 19 GB; **84% (16 GB) lives in `stories.data` JSONB**, almost entirely image bytes — character photos, character avatars, scene reference photos, Visual Bible artifact photos, scene background blobs, debug overlays, and the active page images. The dedicated `story_images` and `story_retry_images` tables hold another 2.5 GB.

Storing image bytes inline in Postgres is wrong for this scale:
- Backups copy 19 GB on every snapshot, most of it untouched binary
- Every story load decompresses TOAST chunks even when the bytes aren't displayed
- Egress is metered through Railway at app-tier prices
- VACUUM FULL takes 5–15 minutes of full-table lock
- Replication lag is awful

After migration:
- DB shrinks to ~500 MB (just metadata + URLs)
- Image storage on R2 = ~$0.15/month for 10 GB
- **Egress from R2 is free.** Cloudflare zero-rates egress, period.
- Backups become trivial
- Story loads return in milliseconds because no TOAST decompression

## What needs to move

| Source | Bytes | Read paths |
|---|---|---|
| `story_images.image_data` | 1.6 GB | `/api/shared/:token/image/:page`, regeneration, repair workflow |
| `story_retry_images.image_data` | 0.9 GB | repair workflow only — diagnostic |
| `style_lab_images.image_data` | 31 MB | dev-mode style-lab UI |
| `stories.data->sceneImages[].imageData` | 97 MB (legacy) | already largely migrated to `story_images` |
| `stories.data->coverImages.*.imageData` | tiny — already migrated | shared viewer cover endpoint |
| `stories.data->characters[].photos` | 448 MB | character avatar generation, scene character refs |
| `stories.data->characters[].avatars` | 408 MB | scene character refs, cover generation |
| `stories.data->visualBible.artifacts/locations` (embedded photos) | ~440 MB | image prompt building |
| `stories.data->styledAvatarGeneration` | 363 MB | dev panel only — could be diagnostic-only |
| `stories.data->coverImages` (residual) | 494 MB | shared viewer |

**Total: ~5 GB of image bytes that should be URLs not blobs.**

## Storage choice: Cloudflare R2

| Provider | Storage / GB / month | Egress |
|---|---|---|
| **Cloudflare R2** | **$0.015** | **$0.00 (zero-rated)** |
| AWS S3 | $0.023 | $0.09/GB |
| Backblaze B2 | $0.005 | $0.01/GB |
| Google Cloud Storage | $0.020 | $0.12/GB |

R2 wins for our access pattern — every shared story view fetches images, every PDF generation pulls the active versions. Egress would dominate any non-R2 choice.

R2 setup:
- Sign up at cloudflare.com (existing CF account preferred — already on the same dashboard as the DNS)
- Create one bucket: `magicalstory-images`
- Create scoped API tokens with R2 read/write
- Add R2 credentials to Railway env vars: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET=magicalstory-images`, `R2_PUBLIC_URL` (custom domain or `https://pub-xxx.r2.dev`)

## Migration phases

### Phase 1: dual-write (low-risk, ~1 day)

Goal: every NEW image written goes to R2; legacy reads still hit Postgres.

1. Add `npm install @aws-sdk/client-s3` (R2 speaks the S3 API)
2. New helper `server/lib/r2.js`:
   - `uploadImage(buf, key) → url`
   - `keyForStoryImage(storyId, imageType, pageNumber, versionIndex) → 'stories/<id>/scene/p<n>/v<v>.jpg'`
   - `keyForCharacterPhoto(charId, slot) → 'characters/<id>/<slot>.jpg'`
3. Schema changes (additive, no drops yet):
   ```sql
   ALTER TABLE story_images ADD COLUMN image_url TEXT;
   ALTER TABLE story_retry_images ADD COLUMN image_url TEXT;
   ALTER TABLE style_lab_images ADD COLUMN image_url TEXT;
   ```
4. Update writers (every place that does `INSERT INTO story_images` or stores `imageData` in JSONB):
   - Upload bytes to R2, get URL
   - Store URL in `image_url` column / JSONB field
   - `image_data` column / `imageData` JSONB field remains for backward compatibility
5. Update readers to prefer URL when present, fall back to bytes.
6. Deploy. New stories have R2 URLs; old stories still have inline bytes.

### Phase 2: backfill (~half day, mostly waiting)

Goal: copy existing inline bytes to R2 in bulk.

1. One-shot script `scripts/migrate-images-to-r2.js`:
   - Reads each `story_images` row in batches of 100
   - Uploads `image_data` to R2 at the canonical key
   - Sets `image_url` and clears `image_data` to NULL in the same UPDATE
   - Logs progress every 100 rows
   - Resumable on crash (skips rows where `image_url IS NOT NULL`)
2. Same script handles inline JSONB images: extracts each `sceneImages[].imageData`, `coverImages.*.imageData`, `characters[].photos`, `characters[].avatars` blob, uploads to R2, replaces with `{"url": "..."}` shape in JSONB.
3. Throttle to ~2 MB/s to avoid hammering production DB.
4. Run during a quiet period.

### Phase 3: cutover + cleanup

1. Verify spot-check: 20 random stories load correctly with all images visible (URL path).
2. Verify regeneration works (it now writes to R2 directly).
3. Verify share-link endpoints return 200 OK and image content matches.
4. Drop the byte columns:
   ```sql
   ALTER TABLE story_images DROP COLUMN image_data;
   ALTER TABLE story_retry_images DROP COLUMN image_data;
   ALTER TABLE style_lab_images DROP COLUMN image_data;
   ```
5. Strip residual `imageData` fields from JSONB.
6. `VACUUM FULL` on `stories`, `story_images`, `story_retry_images`.
7. Final DB size check.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| R2 outage breaks story display | Phase 1 keeps inline bytes as fallback for 1 month after Phase 2 |
| Wrong key naming → bytes uploaded twice | Backfill script keys by stable `(storyId, type, pageNum, version)` — re-runs are idempotent |
| Public bucket leaks images | Use a custom domain + signed URLs OR public read-only with non-guessable keys |
| Failed migration mid-way | Resumable script: only updates rows where `image_url IS NULL` |
| Legacy code reads `image_data` directly | Phase 1 dual-write means both columns are valid simultaneously; deprecate over a release |

## Cost summary

| Item | Today | After migration |
|---|---|---|
| Railway Postgres (storage portion) | ~$5-10/mo for 19 GB | ~$1/mo for ~500 MB |
| Image hosting | n/a (in DB) | ~$0.15/mo on R2 |
| Egress when serving images | Railway-metered | free (R2) |

## Decision points

1. Public R2 access (anyone with the URL can fetch) vs. signed URLs (Cloudflare Worker generates short-lived URLs)? Public is simpler; signed is more secure if any image is sensitive.
2. Do we keep `image_data` columns NULL or drop entirely? Recommend drop after 30-day soak time.
3. Migrate `characters[].photos` and `characters[].avatars` JSONB blobs out of `stories.data` into a separate `character_photos` table at the same time? Cleaner long term.

## Out of scope for this plan

- Image format optimization (WebP, AVIF) — separate follow-up
- CDN caching tuning — R2 is already CDN-backed via Cloudflare
- Image resizing (smaller thumbs, larger high-res for print) — could be added with R2's image-resizing feature

## Estimated total effort

- Phase 1 (dual-write): ~1 day
- Phase 2 (backfill): half day code + ~half day waiting for upload
- Phase 3 (cutover): ~half day
- **Total: 2-2.5 days of focused work**
