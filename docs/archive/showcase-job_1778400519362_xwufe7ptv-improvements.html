Tier 1 — Recurring patterns (high leverage, fix one → many pages improve)

R1. Emma's hair (ponytail + side-part) almost never rendered correctly

Source. Visual Bible: Emma — "ear-length wavy brown hair, side-parted on the left, pulled back into a small ponytail". Fixable issues across pages 1, 2, 5, 6, 9, 10, 11, 12, 13, 14, back cover, front cover, initial page. Entity report flags Emma cells D/F/G as not matching reference R for hair style.

Observation. Two failure modes mixed together:

  1. Hair rendered as down/loose instead of pulled back into a ponytail (most common).
  2. Hair rendered with side-part on the wrong side (right instead of left) — page 1.

This accumulates across the book: even when no single deduction is severe, the character looks subtly different on every page. The composite cover pipeline (now ON) feeds Grok the styled costumed avatar but still drops the ponytail in the redraw pass.

Fix options.

  • (a) Add a "fixed-style" body-shot ref to the per-character grid for hair-critical descriptors. Today the VB grid sends a face crop + body-front. Add a back-of-head / 3/4-back crop showing the ponytail + part. Cheap to compute (one extra avatar variant at character creation), high signal for the model.
    • Pros: deterministic visual anchor, model copies what it sees.
    • Cons: another avatar render at character-creation time (+~$0.02/char + ~5s); needs schema for avatar.backView or similar.
  • (b) Promote hair-style adherence to a dedicated paragraph in scene-expansion + image-generation, not buried inside the character description. Currently hair sits in a long character description blob; Grok/Gemini parse it as "one of many traits". A "hair adherence" instruction would say "Each character's hair style is non-negotiable: copy from the reference exactly, do not invent loose/down hair if the description says ponytail."
    • Pros: zero infra cost; one prompt edit.
    • Cons: prompt-only fix → unreliable; image models routinely ignore one-line "do not" rules in long prompts.
  • (c) Treat "hair style mismatch" as a Grok-cutout repair candidate (face-region inpaint) in the post-gen workflow. The character-repair stage today fixes face/body but doesn't gate on hair specifically.
    • Pros: targeted retroactive fix; uses existing repair infra.
    • Cons: face-region cutouts often miss the ponytail tail (it's outside the bbox); needs bbox expansion logic; runs after-the-fact rather than preventing.

Recommendation: (a) + (b) together. The grid anchor is the durable fix; the prompt note prevents regression in the meantime.

R2. Map artifacts get "Erste/Zweite/Dritte Schatzkarte" baked as visible text

Source. Pages 3 & 4 — fixable issue type=rendered_text, severity MAJOR. Back cover also flags Dritte Schatzkarte missing (different — there it's the artifact itself).

Observation. The artifact's name (in German) is being painted onto the parchment by the model. Story-text overlay then sits on top → double text. Likely cause: the artifact ID/name (Dritte Schatzkarte) is being passed to the image prompt as part of the object descriptor and the model renders "label + object" by reflex.

The plan in tasks/no-too-complicated-we-partitioned-spindle.md already touches image-evaluation.txt §3 (rendered text rule) — but that's about not flagging the cover title. The deeper issue is that image-generation.txt is causing unwanted text by emitting the artifact's name verbatim.

Fix options.

  • (a) Sanitize artifact descriptors before the image prompt. Pass the artifact's visual description ("a folded parchment map with hand-drawn ink lines and a red X") and drop the literal name Dritte Schatzkarte from the prompt entirely.
    • Pros: one-place fix; eliminates the source.
    • Cons: need to thread sanitized descriptors through scene-expansion → image-generation; risk of losing artifact identity in entity-consistency check (which still wants the ID). Solution: sanitize only at the image-prompt boundary.
  • (b) Add an explicit "no rendered text on objects" rule in image-generation.txt. Today the prompt asks for "no text in the image" but only as a soft instruction.
    • Pros: trivial.
    • Cons: same fragility as R1 option (b).
  • (c) Detect baked text post-hoc and inpaint it out. Run an OCR pass on flagged artifacts; if text found, run a Grok inpaint over the text region.
    • Pros: last-line-of-defense, doesn't require pipeline changes.
    • Cons: adds cost + latency on every story; Grok inpaint can leak into the artifact.

Recommendation: (a). The other two are bandages on a self-inflicted prompt.

R3. Noah's costume slips into modern clothes (recurring on pirate-themed pages)

Source. Pages 11, 12, 13 fixable issues: Noah is not wearing the pirate costume. Front cover also: Noah's tan pants instead of tan canvas knee-length shorts. Entity report cell D flags Noah's left sleeve as plaid instead of plain dark green tunic.

Observation. When the story plot puts characters into a costume (here: pirate dress-up), Noah's standard clothing leaks through. This suggests the costumed-avatar reference isn't reaching the per-page generator, or it's being mixed with the standard-clothing description.

server/lib/storyHelpers.js builds the image prompt; check whether costumed-avatar URL (character.avatars.costumed[]) is forwarded to the model when the scene expansion calls for the costume.

Fix options.

  • (a) When coverHint.costumedMode === true (or scene metadata declares costume), route only the costumed-avatar refs to the prompt — strip the standard-clothing description. Today both probably flow through.
    • Pros: removes the contradiction the model is averaging over.
    • Cons: requires a clean costume-mode flag in scene metadata; may need scene-expansion changes to set it.
  • (b) Stronger costume reference: include both costumed-front + costumed-3/4 view in the VB grid for costumed scenes. Same idea as R1(a).
    • Pros: more visual signal.
    • Cons: another avatar render per character per costume.
  • (c) Add a costume-consistency check in entity-consistency-check.txt that explicitly compares against the costumed reference, not the standard.
    • Pros: catches costume drift in the consistency pass.
    • Cons: only fixes detection, not prevention; needs costume metadata to know which reference to compare against.

Recommendation: (a), with (b) once schema supports it.

R4. Gaze direction — characters "look at viewer" instead of at the prop / each other

Source. Back cover: Emma is looking at the viewer instead of fixed downward on the Dritte Schatzkarte. Pages 5, 11, 13: fixable issues with face / gaze problems.

Observation. This is the classic "Grok defaults to camera-facing portrait" failure documented in feedback_grok_inpaint_capability.md (which says Grok can handle facing-direction when given visual identifiers). Yet it still happens — probably because the scene description doesn't explicitly say what each character is looking at.

prompts/scene-expansion.txt could enforce a "gaze target" field per character per scene.

Fix options.

  • (a) Add an explicit gazeTarget field per figure in scene metadata. scene-expansion produces it; image-generation emits "Emma's eyes fixed downward on the open map".
    • Pros: small structural addition; deterministic.
    • Cons: scene-expansion prompt edit + per-scene overhead in tokens.
  • (b) Add a generic image-generation rule: "Characters look at what they are interacting with (the prop, each other, the focal object). Never address the viewer unless the scene explicitly calls for it."
    • Pros: zero structural change.
    • Cons: prompt drift, same fragility as R1(b)/R2(b).
  • (c) Detect "face front-on to camera" in evaluator and auto-route to character-repair. Already partially possible — character-repair handles facing — but needs an evaluator signal.
    • Pros: retroactive guardrail.
    • Cons: another evaluator dimension to specify; deduplication with existing face checks needed.

Recommendation: (a). Per-scene gaze targets are explicit and the model follows explicit visual direction reliably.

R5. Entity grid sometimes paints a totally different person (Noah → elderly man, Daniel → elderly with white mustache)

Source. Entity report:

  • Noah cell D: "completely different character… older man with different facial structure" — affects pages 1,2,3,4,5,7,8,10,11,12,14 (cross-page reference cells)
  • Noah cell G page 8: "elderly man, completely different character"
  • Daniel cell C in pages=-2 (back cover): "elderly man with white hair and a white mustache"

Observation. This is the entity-grid analyzer mis-detecting a face crop on a multi-character page where an elderly background figure (a Hans-like character) gets attributed to Noah/Daniel's grid cell. It's an analysis error, not necessarily a generation error — but it inflates the entity-penalty count and triggers unnecessary repair attempts.

Fix options.

  • (a) Anchor each grid cell to the figure's bbox from the cascade-face-detection stage (which knows which face is which character via avatar similarity). Today the grid analyzer probably picks a face crop heuristically.
    • Pros: addresses root cause; uses existing detection infra.
    • Cons: requires reliable per-character face matching; cost: extra Python service round-trips.
  • (b) Lower the severity of "completely different character" findings when the face-similarity score is below a confidence floor. I.e. distrust the entity report on low-confidence cells.
    • Pros: trivial post-processing of finalChecksReport.entity.
    • Cons: hides real problems too; needs a confidence number from the analyzer (not currently exposed).
  • (c) Run entity consistency on cropped per-character bboxes only, never on the whole page.
    • Pros: removes the "wrong face attributed to character X" failure mode by construction.
    • Cons: significant rework of entity-consistency-check.txt and entityConsistency.js; bbox quality must be high.

Recommendation: (c) is the cleanest long-term answer; (a) is the incremental step toward it.

P10 (page 10). CRITICAL position/scale — Noah at full foreground instead of small/distant

Source. Page 10 fixable: severity CRITICAL, type=position/scale. Noah depicted full-foreground when scene description called for him distant/small.

Observation. The unified prompt encodes position relationally ("on the boat", "behind Roger" — see memory file feedback_position_relational_not_lr.md), but doesn't always encode scale. When the scene wants someone in the deep background, the model needs an explicit "small figure in the distance, ~10% of frame height" type cue, not just "behind".

Fix options.

  • (a) Add a figureScale field per character per scene (foreground/midground/distant) in scene metadata, surfaced in image-generation as "tiny in the distance" / "framed waist-up midground" / "full-frame foreground".
    • Pros: explicit; image models follow scale cues well.
    • Cons: scene-expansion prompt edit; one more JSON field.
  • (b) When a character is described as distant in prose, automatically suppress the per-character avatar reference for that figure (no point sending a 1024px face crop for a 50px figure).
    • Pros: reduces conflict between "high-detail face ref" and "tiny background figure".
    • Cons: bookkeeping.
  • (c) Post-gen detection: if a character bbox covers >40% of the frame on a page where scene metadata says "distant", flag for redo.
    • Pros: retroactive safety net.
    • Cons: another evaluator dimension; bbox detection has to be reliable.

Recommendation: (a). Scale is part of composition and belongs in scene metadata.

  • (c) Implement plan section C (evaluator catches broken VB-referenced objects) so at minimum a redo is triggered.
    • Pros: aligns with existing planned work.

Cross-cutting "what's surprising" (interview takeaways)

  1. One score, one prompt issue, one infra issue dominate this run. R1 (hair) is a single missing reference; R2 (rendered text) is a single sanitization gap; R5 (entity grid mis-attribution) is one analyzer flaw. Three changes would address ~70% of fixable issues.
  2. The composite-cover pipeline shipped but is not retrying low-score covers. attempts=1 on all three covers, including q=0 back cover. The retry/iterate gating that exists for pages does not exist for composite-cover output.
  3. Iterate-round-3 was the best version on multiple pages but final score was still 0. This suggests iterate is finding less bad versions but never breaking out of a wrong-prompt local minimum. Plan section A (single consolidator score across all versions) will at least make the comparison honest, but won't fix the root cause that iterate inherits the same broken prompt.
  4. The unified plan in tasks/no-too-complicated-we-partitioned-spindle.md (cover scoring + cover v0 refs + evaluator object-quality + cover-title carveout) addresses fewer of the patterns than expected. The biggest wins from this run are R1 (hair anchor), R2 (artifact name sanitization), and R5 (per-character bbox attribution) — none of which are in the current plan. Worth weighing before continuing the plan vs. pivoting.

Suggested next-step priorities (if asked)

  1. R2 — Sanitize artifact descriptors (smallest change, eliminates 2 MAJOR issues immediately).
  2. R1(a) — Add back/3-quarter hair-reference avatar variant (one-time character-creation cost, helps every page after).
  3. R5(c) — Per-character bbox-cropped entity consistency (removes phantom "wrong character" findings that inflate penalties).
  4. C2(c) → C2(a) — Audit back-cover landmark wiring, then enable cover retry loop.
  5. R4(a) — Add per-figure gazeTarget (small structural addition, big composition signal).

Each is incremental, none requires a pipeline rewrite, and 1+2+5 together would likely turn this story's 4-cover/14-page run from ~half-bad-pages to ~all-clean on a re-run.
