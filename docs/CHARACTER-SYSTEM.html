1. Data model

One row in characters per character. The data column is JSONB:

{
  id, name, role,
  ageCategory: 'baby' | 'child' | 'teen' | 'adult' | 'elder',
  physicalTraits: { hair, eyes, skin, build, distinguishing },
  physicalTraitsSource: 'photo' | 'manual' | 'ai-extracted',
  clothing: { default: { … }, summer: { … }, winter: { … } },
  changedTraits: { … },          // last-edit deltas
  relationships: { otherCharId: { type, label } },
  photoUrl,                      // R2 — original upload
  faceUrl,                       // R2 — Python-cropped face
  avatars: {
    standardUrl, summerUrl, winterUrl,        // 2×2 grids per outfit
    bodyThumbnailsUrl: { default, summer, … },
    faceThumbnailsUrl: { default, summer, … },
    styledAvatars: { [artStyleId]: { standard, summer, winter } } // inline base64 for now
  }
}

2. Photo upload & analysis

  1. Wizard step 2 captures a photo via webcam or file upload.
  2. POST /api/analyze-photo sends bytes to the Python service.
  3. photo_analyzer.py runs MediaPipe / MTCNN / Haar cascade for face detection and rembg for background removal.
  4. Result: face bbox, cropped face PNG (transparent bg), and quality flags (blur, occlusion, multi-face).
  5. Frontend may show FaceSelectionModal if multiple faces are detected.
The Python service runs on port 5000 and must be running locally for photo upload to work in dev. Production runs it as a Railway service.

3. Trait extraction & sourcing

Traits can come from three sources, tracked in physicalTraitsSource:

SourceMechanismUsed when
photoGemini vision call on the cropped face / bodyUser uploaded a photo and accepted the auto-extracted traits.
manualUser typed each traitNo photo, or user overrode the extracted values.
ai-extractedSame Gemini call but uneditedDefault for photo flow until the user edits a field.

Extracted traits include hair (colour, length, style), eyes (colour, shape), skin tone, build, distinguishing features (freckles, glasses, scar). Stored under physicalTraits and used both in the avatar prompt and as text in the story image prompt for non-photo characters.

4. Clothing slots

Each character has multiple clothing slots; the active slot per page depends on season, story context, or explicit overrides:

  • default — everyday outfit; populated by trait-extraction or user input.
  • summer — light/short variants; auto-derived if not explicitly set.
  • winter — coat/jacket variants; auto-derived if not explicitly set.
  • Custom slots — period costumes (e.g. medieval for Wilhelm Tell stories).

Each slot has its own avatar (face + body crops) and 2×2 grid. resolvedClothing in the scene prompt picks the right slot based on scene metadata.

5. Avatar generation

Avatars are how the image model recognises the character. Each clothing slot gets a 2×2 grid (face-front, face-profile, body-front, body-profile) plus separate face and body crops.

Default backend

Grok Imagine is the default avatar backend. Gemini fallback exists. Configured in server/config/models.js; can be overridden in dev mode via useDeveloperMode.

Quality gate

Generated avatars are scored by prompts/avatar-evaluation.txt. finalScore = MIN(face, age, clothing) — adult-rendered-as-child or wrong-clothing avatars trip the gate even if face match is high. Failed avatars persist status='failed' so the wizard pre-flight skips them.

ACE++ for face consistency

For face-only consistency at low cost, Runware's ACE++ (server/lib/runware.js) can re-skin a generated body with the user's actual face. Used selectively — quality is lower than Grok but cheap.

6. Storage shape (R2)

R2 keys (built in server/lib/r2.js):

characters/{characterId}/photo.jpg
characters/{characterId}/face.png
characters/{characterId}/avatars/{slot}/standard.jpg
characters/{characterId}/avatars/{slot}/body.jpg
characters/{characterId}/avatars/{slot}/face.jpg
characters/{characterId}/avatars/styled/{artStyleId}/{slot}.jpg  (Phase 1e — pending)
stories/{storyId}/vb/{entityId}.jpg
stories/{storyId}/page-{pageNum}/v{n}.jpg

URL fields (standardUrl, bodyThumbnailsUrl, …) live alongside the original inline base64 fields during the dual-write phase. loadAvatarBytes(avatar, slot) in server/lib/characterPhotos.js abstracts the lookup — readers prefer URL, fall back to inline.

7. Shot-aware reference packing

Story image prompts pack character references into Grok / Gemini's reference slots. The pack shape depends on the scene's shot type:

ShotRefs per characterReason
wide1 (body crop)Face is too small to matter; body crop preserves clothing + posture.
medium / close-up2 (face crop + body crop)Both face features and clothing matter at this distance.
fallback (no variantUrls)1 (full 2×2 grid)Older characters without crop variants — uses standard avatar.

Implementation: applyReferenceMode in server/lib/storyHelpers.js reads sceneMetadata.shot, splits each character's variantUrls.face and variantUrls.body into separate refs.

8. Visual Bible role

Each story snapshots its characters into the Visual Bible (visualBible.characters) — assigning stable IDs (CHR###) used throughout the pipeline. The grid + linker (server/lib/visualBible.js) ensures that "the boy in the red shirt" on page 3 is the same entity as "Noah" on page 7.

The VB also contains objects (props with ART### IDs) and locations (LOC### IDs). Locations link to historical_locations rows when applicable — see landmarks.
