2. IP-based detection

GET /api/user/location in server/routes/user.js. Reads forwarded headers (x-forwarded-for, cf-connecting-ip, x-real-ip) to find the original client IP, calls ip-api.com (free, 45 req/min), returns { city, region, country }. Returns nulls for localhost / private IPs.

Used by the wizard to pre-fill the location field; not authoritative — the user can edit or clear it.

3. User-entered + Nominatim verify

POST /api/user/verify-location body: { city?, plz?, country? } (country defaults to Switzerland). Calls Nominatim (OpenStreetMap), 24h in-memory cache.

Response: { verified, city, country, plz, displayName, lat, lon }.

Wizard UX

  • Country is pre-filled to Schweiz / Suisse / Switzerland based on the UI language.
  • 4-digit Swiss PLZ input — entering only PLZ resolves the city via Nominatim and back-fills the city input.
  • Debounced (500 ms) verify on every keystroke; ✓ (green check), ✗ (red x), or spinner.
  • On not-found: explicit error message with a "Continue without location" action that clears the field.
  • OK button disabled while a check is in flight.

Frontend: client/src/pages/wizard/WizardStep3BookSettings.tsx. Service: storyService.verifyLocation.

4. Landmark discovery

Once a location is set, the frontend fires POST /api/landmarks/discover (fire-and-forget) so landmark data is ready by the time story generation starts.

Backend (server.js):

  1. Check landmark_index table for cached rows for the city.
  2. Check the in-memory cache (LANDMARK_CACHE_TTL).
  3. Call discoverLandmarksForLocation(city, country) — Wikipedia/Wikidata geosearch + categorisation. Implementation: server/lib/landmarkPhotos.js.
  4. Stash results in both caches.

Discovery returns landmark name, lat/lon, Wikipedia summary, and category (mountain / castle / square / monument). The story pipeline picks 0-3 landmarks per story based on the outline.

5. Curated historical_locations

For period stories, generic Wikipedia data is too thin. historical_locations rows hold:

  • Stable id (e.g. LOC_TELL_ALTDORF_PLATZ) and dbKey slug.
  • Display name in DE/FR/IT/EN.
  • Period (e.g. "13th century").
  • Reference photo (R2 URL) — typically a curated Grok-generated period view.
  • Linked story type / theme.

Companion table historical_objects for period props (hat, crossbow, spear). Same shape, parallel use.

Curation scripts live in scripts/admin/curate-tell-locations.js, refine-tell-vN.js (40+ iterative refinement rounds), generate-tell-locations.js.

6. Visual Bible injection

When the unified Claude pass produces an outline, the linker (server/lib/visualBible.js → injectHistoricalLocations) matches outline-mentioned places against historical_locations by:

  1. existing.dbKey === loc.dbKey (preferred — stable slug match).
  2. Name fuzzy match (fallback).

Matched locations are added to the Visual Bible with id: LOC###, the curated reference photo, and dbKey. Scenes whose hint.objects contains the LOC ID get the location injected into the image prompt as PRE-POPULATED LOCATIONS, with rules forbidding generic substitution.

Reference packing (server/lib/grok.js → packReferences) places landmark slot at position 1 (before character slots). Accepts both raw base64 and data URI for landmark bytes.

7. Wilhelm Tell case study

Wilhelm Tell is the first historical theme with full curated coverage. Curated assets:

  • ~7 historical_locations: Altdorf marketplace, Bürglen, Hohle Gasse, Tellsplatte, Lake Lucerne (storm), Bristen view, Altdorf church.
  • ~3 historical_objects: medieval crossbow, Gessler's hat, Tell's hat.

The historical-guides prompt (prompts/historical-guides.txt) carries the Tell-specific guidance. Each location went through 5-30 Grok refinement passes to land on a usable reference photo. See scripts/admin/refine-tell-v40.js for the latest refinement scaffolding.
