Image Generation Methods

Inventory of every image-generation entry point in the codebase, the provider it calls, and the situations it's used for. Use this when you need to know which function to call for a new feature, or why a specific path uses one model and not another. The defaults in server/config/models.js are the source of truth; this page mirrors them.

2. Providers

ProviderEndpoint we callStrengthsWeaknesses
Grok Imagine (xAI Aurora) /v1/images/edits (multi-ref) & /v1/images/generations Fast (2–4× Gemini), cheap ($0.02), accepts up to 3 reference images on Standard, scene-composite-friendly. Style consistency across pages is the weakest of any model we use. Identity preservation on the 2×2 avatar sheet is weak. No negative prompts. Output aspect = input image aspect — the aspect_ratio param is silently ignored on edit. Standard model: max 3 refs. Pro model: max 1 ref (we stitch).
Gemini Imagen / 2.5 Flash Image generateContent with image part Strongest prompt adherence, most consistent style across pages, best at structured illustration / watercolor / whimsical, handles complex scenes well. $0.04/image (double Grok). Adult-face photos trigger IMAGE_OTHER safety refusals — unusable for avatar generation from real user photos. 30 KB prompt limit (vs Grok's 7.5 KB) is the only context where it has more room than Grok.
Runware FLUX (Schnell / Dev) Runware HTTP API Ultra-cheap ($0.0006–$0.004/image), good for dev-mode A/B testing. Schnell is 4-step, very fast. 3 KB prompt limit. No multi-image reference (only IP-Adapter style hints). Not used in production page/cover generation.
Runware ACE++ Runware HTTP API (ace-plus-plus model) Face-consistent generation: reference image's face survives the prompt change. Used as the secondary avatar backend for explicit identity-preservation requests. $0.005/image. Slower than Grok. The output composition is harder to control via prompt — best for "same face, different scene" tasks, not for "render this prose precisely".

5. Aspect-ratio behaviour by provider

This trips people up when switching providers. The output aspect is determined differently per provider:

ProviderHow output aspect is determinedWhat our wrapper does
Grok edit Matches the input image's aspect. The API's aspect_ratio param is silently ignored on edit calls. editWithGrok coerces every input image to the requested aspectRatio before the API call. padInput: false (default) crops the longer axis (fit:cover); padInput: true pads the shorter axis with white (fit:contain). Use padInput for avatar inputs that are already on a white background.
Grok generate (text-to-image) Honours the aspect_ratio param. generateWithGrok passes it through directly.
Gemini imagen Honours imageConfig.aspectRatio in the request body. callGeminiAPIForImage passes it through. Output exactly matches.
Runware Width × height in pixels (not a ratio string). Caller computes width and height from the desired aspect.

This is the single most common cause of "the new model produces weird shapes" — switching from Gemini to Grok without realising the wrapper now crops the input. See tests/avatar-debug/ for a worked example.

6. Dispatch & override layers

Three places control which model runs for a given image:

  1. Global defaults — server/config/models.js → MODEL_DEFAULTS. Change here to retune the whole pipeline.
  2. Per-story override — values set on the story_jobs.input_data blob at creation time (imageBackend, imageModel, scoreModel, etc.) snapshot the defaults so re-runs of the same story behave identically.
  3. Per-call override — every generation function accepts imageBackendOverride / imageModelOverride kwargs. Used by the dev-mode "Test Models" and "Style Transfer" UIs to bypass the configured default for one image.

Auth: the dev-mode UIs route through the admin JWT path; the per-call overrides only fire when the request has an admin token. End-users always get the default config.

7. Why-not table

Approaches we tried, evaluated, and dropped — so future "let's just use X" suggestions can be checked against actual evidence.

ApproachStatusOne-line reason
FLUX Dev / Schnell for pages❌ rejectedIdentity preservation across characters is unusable for storybook pages.
Gemini for clothing avatars❌ rejectedIMAGE_OTHER safety filter rejects adult-face inputs, leaving avatars permanently status='pending'.
ACE++ as default avatar backend🟡 conditionalLocks identity well, but composition control is weaker than Grok edit. Used only as the explicit-opt-in route.
Single Gemini call for everything❌ rejectedCost (2× Grok) and the IMAGE_OTHER issue make it unusable for the avatar path even if scenes still suited it.
Grok Pro everywhere🟡 conditional$0.07/image (3.5× Standard). Reserved for situations where Standard's quality is genuinely insufficient and a single regen will save more than the price difference.
Single-portrait avatars (no 2×2 grid)🟡 candidate3 calls/character instead of 1, but each gets full attention → better identity. Open lever, not yet shipped.
Kontext LoRA for character locking🟡 conditional1 training image, $2–5 per character, 99% identity. Too expensive for the current pricing model; revisit if a tier supports it.
