Image Generation Methods

Inventory of every image-generation entry point in the codebase, the provider it calls, and the situations it's used for. Use this when you need to know which function to call for a new feature, or why a specific path uses one model and not another. The defaults in server/config/models.js are the source of truth; this page mirrors them.

2. Providers

ProviderEndpoint we callStrengthsWeaknesses
Grok Imagine (xAI Aurora) /v1/images/edits (multi-ref) & /v1/images/generations Fast (2–4× Gemini), cheap ($0.02), accepts up to 3 reference images on Standard, scene-composite-friendly. Style consistency across pages is the weakest of any model we use. Identity preservation on the 2×2 avatar sheet is weak. No negative prompts. Output aspect = input image aspect — the aspect_ratio param is silently ignored on edit. Standard model: max 3 refs. Pro model: max 1 ref (we stitch).
Gemini Imagen / 2.5 Flash Image generateContent with image part Strongest prompt adherence, most consistent style across pages, best at structured illustration / watercolor / whimsical, handles complex scenes well. $0.04/image (double Grok). Adult-face photos trigger IMAGE_OTHER safety refusals — unusable for avatar generation from real user photos. 30 KB prompt limit (vs Grok's 7.5 KB) is the only context where it has more room than Grok.
Runware FLUX (Schnell / Dev) Runware HTTP API Ultra-cheap ($0.0006–$0.004/image), good for dev-mode A/B testing. Schnell is 4-step, very fast. 3 KB prompt limit. No multi-image reference (only IP-Adapter style hints). Not used in production page/cover generation.
Runware ACE++ Runware HTTP API (ace-plus-plus model) Face-consistent generation: reference image's face survives the prompt change. Used as the secondary avatar backend for explicit identity-preservation requests. $0.005/image. Slower than Grok. The output composition is harder to control via prompt — best for "same face, different scene" tasks, not for "render this prose precisely".

5. Aspect-ratio behaviour by provider

This trips people up when switching providers. The output aspect is determined differently per provider:

ProviderHow output aspect is determinedWhat our wrapper does
Grok edit Matches the input image's aspect. The API's aspect_ratio param is silently ignored on edit calls. editWithGrok coerces every input image to the requested aspectRatio before the API call. padInput: false (default) crops the longer axis (fit:cover); padInput: true pads the shorter axis with white (fit:contain). Use padInput for avatar inputs that are already on a white background.
Grok generate (text-to-image) Honours the aspect_ratio param. generateWithGrok passes it through directly.
Gemini imagen Honours imageConfig.aspectRatio in the request body. callGeminiAPIForImage passes it through. Output exactly matches.
Runware Width × height in pixels (not a ratio string). Caller computes width and height from the desired aspect.

This is the single most common cause of "the new model produces weird shapes" — switching from Gemini to Grok without realising the wrapper now crops the input. See tests/avatar-debug/ for a worked example.

6. Dispatch & override layers

Three places control which model runs for a given image:

  1. Global defaults — server/config/models.js → MODEL_DEFAULTS. Change here to retune the whole pipeline.
  2. Per-story override — values set on the story_jobs.input_data blob at creation time (imageBackend, imageModel, scoreModel, etc.) snapshot the defaults so re-runs of the same story behave identically.
  3. Per-call override — every generation function accepts imageBackendOverride / imageModelOverride kwargs. Used by the dev-mode "Test Models" and "Style Transfer" UIs to bypass the configured default for one image.

Auth: the dev-mode UIs route through the admin JWT path; the per-call overrides only fire when the request has an admin token. End-users always get the default config.

7. Routing — when each method works best

Three orthogonal dispatch dimensions decide what runs for any given page. Today only one of them (scene complexity) ramifies fully; the others have proposed but un-shipped rules captured here as the next routing iteration.

Dimension A — Direct vs Composite (by cast size + interaction)

Why it matters: direct path is one Grok call (~$0.02). Composite is three Grok calls (~$0.06) + optional phantom-pose render (+~$0.02 per cast character — so a 6-cast composite page with phantom-pose is ~9 calls / ~$0.18). Composite buys structural control (position, depth, pose) at the cost of latency, $/page, and accumulated style drift across 3-4 sequential edits.

"Cast" vs "background figures": "cast" means named characters in sceneCharacters[] who must look like their reference photo. Anonymous background extras described inline ("a crowd of villagers in the marketplace", "two guards by the gate") are not cast — they go into emptyScenePrompt and are painted as part of the background. They do NOT drive composite routing.

"Interaction" means characters acting on each other — chasing, fighting, embracing, pulling a rope between them, one shooting at another, handing an object across. Two named characters standing side-by-side, or one in foreground and another in background not interacting, is NOT an interaction scene.

Cast (named)Interaction?Recommended pathWhy
0n/aDirect, text-only Grok generate, refMode offNo identity to preserve.
1n/aDirect, refMode looseOne subject, one reference. Grok handles single-character identity well in one pass.
2anyDirectTwo refs fit Grok's attention budget. Even when they interact, two-character direct generations are usually clean.
3anyDirectThree refs still within Grok's reliable range — costume drift creeps in but identity holds. Composite's overhead doesn't pay off yet.
4+anyComposite (3-pass) — phantom-pose render recommended at 5+Direct's attention divides too thin across 4+ named characters; faces and costumes start swapping. Composite stages BG → blocking → blend, locking each character's position and pose separately.

The "interaction" column is currently advisory, not a separate decision branch. The simpler rule "cast ≥ 4 → composite, cast ≤ 3 → direct" works for ~95% of cases. Interaction matters most as a tiebreaker for 3-cast pages where direct sometimes fumbles a wrestling / hand-off / pull-rope scene; the per-page test panel (see below) is the right tool for those edge cases.

Current implementation: the cast-aware dispatcher is live in server/lib/imageRouter.js (decidePageRoute), called from server.js before each page job. Routes are: cast 0 → direct, refMode off; cast 1–3 → direct, refMode loose; cast 4–5 → composite; cast 6+ → composite + phantom-pose. Hard overrides via inputData.composite / inputData.phantomPoseRender still bypass the cast count.

Per-image rerun: in admin/dev mode, every page exposes a "Test Models" panel (client/src/components/generation/TestModelsPanel.tsx → POST /api/stories/:id/test-models/:pageNumber at server/routes/regeneration.js). It picks model × composite-on/off × refMode × phantom-pose × populated-plate-source and rebuilds just that image. Outline and scene-expansion are reused verbatim, so a single experiment costs only the image-gen calls of the chosen method. See covers-and-composite §4 for the per-method cost table.

Dimension B — Reference mode (identity strictness)

Set via MODEL_DEFAULTS.referenceMode (current default: 'loose'). Controls what reference images are attached to a direct-path page generation. Looser modes trade identity stability for painterly cohesion in the output.

ModeWhat's attachedBest for
strictBody-no-bg photos + costumed avatars + styled avatars + full VB gridHero pages with named characters who must look recognisably like the user. Highest identity preservation; sometimes too tight, output looks photographic rather than illustrated.
loose (default)Costumed + styled; photo only on close-ups; identity cells without grid frameMost pages. Identity stays consistent across pages because styled avatars carry the look forward; output reads as illustration, not photo.
styled-onlyOnly the in-art-style avatar; identity cells without grid frameStylised pages where photographic identity would clash with the art style (e.g. stylised fairy-tale opener).
offNo character refs, no VB grid (text-only)Empty scenes / pure environment shots. Faster, cheaper, and avoids "phantom character" hallucinations from refs that aren't relevant.

Dimension C — Per-page model (scene complexity)

Set via MODEL_DEFAULTS.sceneRouting and the scene metadata's sceneComplexity field (Sonnet emits this during outline expansion). Today both simplePageImage and complexPageImage default to grok-imagine, so this dimension is dormant — but the wiring exists to route complex scenes to a different model if we want to A/B it.

sceneRouting valueBehaviour
'auto' (default)Reads sceneMetadata.sceneComplexity ('simple' | 'complex'). Routes to simplePageImage or complexPageImage respectively.
'grok'Force Grok for every page.
'gemini'Force Gemini for every page (only useful as a comparison run; not recommended for production after the Gemini→Grok switch).
any model idForce the named model for every page.

Composing all three dimensions

The current dispatcher (server.js, story-job processing) evaluates them in this order:

  1. Composite vs direct (Dimension A) — currently a global flag (enableSceneComposite) + per-story opt-out.
  2. Phantom-pose render (sub-flag of composite) — MODEL_DEFAULTS.phantomPoseRender or per-story override. Default off.
  3. Reference mode (Dimension B) — only affects the direct path. Composite's blocking + blend pass have their own reference shape (VB grid in Image 2 of blend).
  4. Per-page model (Dimension C) — only affects the direct path. Composite always runs through Grok edit.

Next roadmap step on routing: replace the global enableSceneComposite with a cast-size-aware dispatcher that picks direct vs composite per page based on the resolved cast count + complexity flag, per the table at the top of this section.
