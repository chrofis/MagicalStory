Key server/lib/ files

  • images.js — image generation, quality eval, repair pipeline, Visual Bible grid.
  • storyHelpers.js — scene metadata, prompt building, character composition.
  • visualBible.js — entity tracking, location injection.
  • scaleRepair.js — relocate-misplaced-character workflow.
  • entityConsistency.js — bbox detection, face-merge, object canonicalisation.
  • textRegion.js — calmness map + white wash for text overlay (see text overlay).
  • characterPhotos.js — avatar/VB byte loading (R2-aware).
  • grok.js, runware.js, textModels.js — vendor adapters.
  • pdf.js, coverIterate.js, feedbackConsolidator.js, repairLogic.js — pipeline-specific utilities.
  • r2.js — Cloudflare R2 client + key helpers.

Cloudflare R2

R2 holds the binary payloads that used to bloat characters.data / stories.data. Roughly 1500× cheaper per GB-month than Postgres. Key helpers live in server/lib/r2.js; reader helpers loadAvatarBytes / loadVbReferenceBytes in server/lib/characterPhotos.js abstract URL-vs-inline lookups.

Migration history: characters and VB references migrated in Phases 0–1 (writers + readers); Phase 2 (rewire all readers) and Phase 3 (drop inline base64) remain. See archive for the original storage proposal.

6. Deploy topology

  • Production: master branch → Railway → magicalstory.ch. Each push auto-deploys, kills any in-flight story generation, and causes brief downtime.
  • Staging: staging branch → Railway → staging.magicalstory.ch. Same pipeline, separate Postgres + R2 bucket, optional HTTP Basic Auth gate.
  • Branch flow: feature/X → PR → merge into staging → smoke-test on staging → ask for explicit per-push approval → push to master.

Full deploy / staging / email setup is in operations.
