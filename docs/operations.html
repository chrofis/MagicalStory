1. Deploy flow

Never push to master without explicit per-push approval. Each prod deploy auto-runs Railway, kills any in-flight story generation (real users mid-creation lose their work), and causes brief downtime. "Functionally safe", "additive", "trivial typo fix" are not valid reasons to skip the approval step — they all have the same downtime + in-flight-job cost.

Mandatory flow for any change:

  1. git push origin staging (or push a feature branch + PR-merge into staging).
  2. Verify on https://staging.magicalstory.ch — page loads, smoke-test the affected area.
  3. Ask the user explicitly: "OK to push to master?"
  4. Only after an explicit yes (not silence, not "looks good", not earlier authorisation for a different change): git push origin master.

Each prod push is its own approval moment.

2. Staging environment

One-time provisioning (already done; documented for reference):

  • Railway service — separate from prod, autodeploys from staging branch.
  • Postgres — separate database, migrations run on first boot via server/services/database.js.
  • R2 bucket — separate bucket, Cloudflare DNS for images.staging.magicalstory.ch.
  • Cloudflare DNS — staging.magicalstory.ch CNAME → Railway-provided host.
  • Stripe — test-mode keys; orders never charge.
  • Env vars — same set as prod with BASE_URL=https://staging.magicalstory.ch and the staging Postgres / R2 credentials.

3. Staging Basic-Auth gate

Optional HTTP Basic Auth gate set via STAGING_AUTH_USER / STAGING_AUTH_PASSWORD in server.js. Bypassed for:

  • All /api/* routes (browsers don't replay Basic Auth on XHR reliably).
  • PWA / SEO statics: /manifest.json, /robots.txt, /sitemap.xml, favicons, og-image*.

The gate prevents random Google indexing of staging while keeping login / register / static / SEO infrastructure functional.

4. Email (Resend)

Resend is the transactional email provider. Configured via RESEND_API_KEY, EMAIL_FROM, EMAIL_REPLY_TO. Implementation: email.js at the project root.

Email types in use:

  • Email verification (after signup or trial → full upgrade).
  • Password reset.
  • Order confirmation (post-payment).
  • Print status updates (Gelato webhook → email).
  • Story-ready notification (when a long generation completes after the user closed the tab).

Templates are React Email components; the react-email skill is the recommended workflow when adding a new template.

5. Demo / showcase command

The orchestrator scripts/admin/showcase.js (alias npm run showcase) runs an end-to-end demo story with a fresh account, real characters, and real photos.

What it does

  1. Picks a rotation entry (default = next from tests/demo-rotation-state.json; override via --entry=N).
  2. Creates a fresh timestamped account: demo-{family}-{YYYYMMDD-HHmm}@magicalstory.ch.
  3. Uploads characters + curated photos from tests/fixtures/demo-photos/{family}/.
  4. Triggers a Playwright spec (tests/demo-story.spec.ts) that drives the wizard and starts story generation.

Each run is fully isolated — old stories stay accessible on their original accounts. Default backend is production; for local: npm run showcase:local; for staging: SHOWCASE_API_BASE=https://staging.magicalstory.ch npm run showcase.

Family rotation

tests/helpers/demo-rotation.json lists the demo families (Bergers, Millers, …) with index. The user might say "showcase the Bergers" → look up the matching index and pass --entry=N.

Generating photos

If photos for a family don't exist on disk: node scripts/admin/generate-demo-photos.js --family=<id> --save-to=true --no-upload. Inspect, then proceed.

Demo family portraits are locked. Do not regenerate generate-demo-photos.js on approved files unless explicitly asked for that specific character.

6. Log analysis

scripts/analyze-story-log.js parses Railway log dumps:

node scripts/analyze-story-log.js                           # Latest log in ~/Downloads
node scripts/analyze-story-log.js ~/Downloads/logs.X.log   # Specific log file

Output: story info (title, language, characters, pages), timing per stage, costs per provider, errors / warnings / fallbacks / low scores.

Timezone: Railway timestamps are UTC; the user lives in Switzerland (CEST = UTC+2 in summer, CET = UTC+1 in winter). Always translate before correlating with user-mentioned times. Quote both when reporting: "23:50 UTC (01:50 CEST)".

7. Database migrations

Migrations are file-based (committed in server/migrations/) and run on boot from server/services/database.js. Each migration is idempotent (CREATE TABLE IF NOT EXISTS, ALTER TABLE … ADD COLUMN IF NOT EXISTS, etc.).

Adding a new migration:

  1. Drop a new NNN_name.sql in server/migrations/.
  2. Make every statement idempotent — staging may run it on a half-migrated DB.
  3. Verify on staging first; staging Postgres started life partially divergent from prod.
  4. scripts/admin/compare-schemas.js diffs prod vs staging schemas to catch drift.

Long-standing gotcha: story_images was created manually on prod years ago and only had ALTERs in code, not CREATE. A fresh DB (e.g. staging) needed an explicit CREATE TABLE IF NOT EXISTS story_images (now in place).
