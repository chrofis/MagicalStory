The SCENE_HINT and IMAGE_PROMPT are the ONLY source of truth for which characters belong in the illustration. The STORY_TEXT is background narrative that frequently mentions characters who appear on OTHER pages, or who are referenced in dialogue without being physically present in this scene. A character mentioned in STORY_TEXT but absent from SCENE_HINT and IMAGE_PROMPT is NOT missing and MUST NOT be penalized — under any check, in any step, in any scoring decision below. This rule overrides every other rule in this prompt that could be read as conflicting.

**STEP 3: CHECK THE STORY BEAT**

Only check these things (using SCENE_HINT + IMAGE_PROMPT as the source of truth). Each check below carries a default severity tag in [brackets] — apply that tag to the `semantic_issues` entry when the check fails. The tag drives a separate quality penalty (CRITICAL = −30, MAJOR = −20, MODERATE = −10). Be conservative — if unsure between two levels, pick the lower.

- Correct main action? (giving ≠ receiving, hugging ≠ fighting) — **[CRITICAL]** if the rendered action is the wrong beat
- Correct number of named characters? — **[CRITICAL]** if a named character is missing; **[MAJOR]** if a stand-in figure is rendered in their place
- Correct setting/location? — **[CRITICAL]** if completely wrong location; **[MAJOR]** if wrong sub-area within the right location
- Correct surface? (on island, not in water; on path, not in river) — **[MAJOR]**
- Correct broad clothing category? (medieval ≠ modern, winter coat ≠ t-shirt, costume ≠ casual) — **[MAJOR]**
- Correct proportions / age band? When the SCENE_HINT or IMAGE_PROMPT explicitly names a band — "grade-school", "teenage", "adult", "senior" — or describes proportions ("clearly taller and leaner than grade-schoolers", "rounded grade-school proportions", "adult bone structure"), flag if the rendered figure's proportions clearly belong to a different band (e.g. prose says "teenage" but figure has grade-school proportions, or "adult" but figure has teenage proportions). This is a narrative-consistency check on body proportions only — within-band variation does not count, and you are NOT judging the character's own attributes, just whether the prose-named band matches the rendered band. — **[MAJOR]**
- Key objects present? (an object named in DECLARED INTERACTIONS, or named ≥ 2 times in SCENE_HINT/IMAGE_PROMPT prose, is "key") — **[MAJOR]** per missing key object
- Character-object plausibility? (character "at a table" must be beside it, not merged/sunk into the surface; character "on a bridge" must stand on it, not clip through it) — **[MAJOR]**
- Correct weather/lighting? (night = dark, snow = snow) — **[MODERATE]**
- Emotion matches the scene? (smiling during danger = wrong, scared during calm = wrong) — **[MODERATE]**. If you notice ANY emotion/tone defect — wrong facial expression for the narrative beat (broad smile when the scene's prose describes urgency, fear, sadness, or shouting; vacant calm during a chase or shock; placid happiness during a crisis), wrong body language (relaxed posture in declared danger, tense posture in declared calm), or any mismatch between a character's rendered affect and the action the STORY_TEXT describes — you **MUST** emit it as a structured `semantic_issues[]` entry: `{problem: "<character> shows <observed emotion>, scene expects <expected emotion> (STORY_TEXT says ...)", severity: "MODERATE"}`. Do NOT mention emotion/tone defects only in `issues_summary` prose — the downstream repair pipeline ONLY reads `semantic_issues[]`. Subtle expression deltas already excluded above (calm vs thoughtful, slightly worried vs neutral) remain non-deductions.
- Gaze direction matches a DECLARED INTERACTION? Only flag when two characters listed in DECLARED INTERACTIONS as interacting face away from each other, or a character listed as watching a specific named object is looking elsewhere. Prose phrases like "looking toward the distance", "gaze upward", "facing the sky", "scanning sideways" are NOT declared interactions — do NOT flag. — **[MODERATE]** when flagged
- **Declared interactions placement** (overrides the "ignore spatial arrangement" rule below — declared interactions ARE spatial requirements that matter):
  - Verify each entry: the object is at that exact location relative to the character. "In pocket" must be a tiny silhouette in the pocket, not a held plush. "Climbing rope" must show hands gripping the rope. "Standing on stairs" must show feet on the step.
  - Categories that count as wrong: held-vs-pocket, on-head-vs-in-hand, gripping-vs-near, on-shoulder-vs-in-hand. Flag as `wrong_interaction` — **[MAJOR]**.
  - Pose details from the scene prose that are NOT in DECLARED INTERACTIONS (e.g. "hands drawn in tight", "fingers curled", "chin tucked") are descriptive only — do NOT flag.
  - Extra background figures in scenes whose hint describes a populated setting (crowd, vendors, villagers, marketgoers) are scene-appropriate — do NOT flag.
- **Left/right mirror is always equivalent — never flag.** Any body-part swap between left and right (left arm vs right arm, left hand vs right hand, left foot vs right foot, left pocket vs right pocket, left shoulder vs right shoulder) is a mirror, not a defect. Compound mirrors count too — if the declared interaction says "right foot forward, left arm swept back" and the image shows "left foot forward, right arm swept back", that is the same pose reflected. Do NOT raise a `wrong_interaction` issue for any mirror case.

**Ignore everything else.** The image prompt contains detailed physical descriptions from reference photos — minor visual differences are expected. Specifically ignore:
- Character identity, names, recognizability. A separate system with reference photos handles identity.
- Exact colors of ANY element — clothing, objects, vehicles, hair. Only the broad category matters (modern vs medieval, red vs blue). Shade variants (lilac vs purple, crimson vs red, light-blue vs blue) never deduct.
- Clothing details: patterns, fabric type, accessories (backpack present/absent), specific garment (hoodie vs sweatshirt).
- Hair details: exact length, exact shade, ponytail vs loose, style.
- Face details: jaw shape, nose shape, exact features.
- Background characters or extras.
- Extra objects beyond what the story beat requires.
- Camera angle, shot type, framing, perspective.
- Subtle expression differences (thoughtful vs focused, slightly worried vs neutral).
- Exact pose, gesture, body language.
- Motion verbs in the scene prose (walking, running, approaching, scanning). A still image cannot show motion; any frozen pose satisfies them.
- Internal-state phrases ("appears thoughtful", "looks distracted").
- Left/right positioning, spatial arrangement, depth labels (foreground/midground/background). Position strings like "left foreground" or "right midground" are metadata hints, not requirements.

**SCORING**

Start at 10. Only these deductions exist:
- Missing character (listed in SCENE_HINT or IMAGE_PROMPT — depth label "foreground/midground/background" is not part of the requirement): -3
- Character merged into or clipped through an object (AI compositing artifact): -3
- Declared interaction missing or in a genuinely different category (e.g. object in pocket drawn as large held plush, on-head drawn in-hand): -3. Any left/right mirror (arm, hand, foot, shoulder, pocket, or compound body-part swap) is equivalent and does NOT deduct — score as if the interaction matched. Pose details in the scene prose that are NOT in DECLARED INTERACTIONS do NOT deduct.
- Wrong setting/location: -2
- Wrong main action: -2
- Missing key object: -2
- Wrong emotion (happy in danger, scared in calm scene): -1
- Wrong gaze (two characters listed in DECLARED INTERACTIONS facing away from each other, OR character listed as watching a specific named object looking elsewhere): -1
- Wrong clothing type on a character: -1
- Wrong proportions / age band for a named character (grade-school / teenage / adult / senior mismatch): -2

No other deductions. Extra objects, background people, non-declared spatial arrangement, characters from STORY_TEXT not in SCENE_HINT — all score 10.

VERDICT: PASS (7+), FAIL (0-6)

**OUTPUT FORMAT**

Return JSON only. Put every issue you find in the `semantic_issues` array. Do not use any other field name for issues — the parser reads `semantic_issues` and discards anything else.

**Output rules (strict):**
- The `score` field must be an integer 0–10 — no decimals, no fractions.
- The `verdict` field must be exactly "PASS" or "FAIL" (uppercase, no other values).
- Do not add explanatory text before or after the JSON. Plain JSON only — markdown code fences are tolerated by the parser but unnecessary.
- **Do NOT emit `issues_summary`, `notes`, `comments`, or any other prose field.** Every observation — including emotion/tone mismatches, posture mismatches, gaze defects — goes in `semantic_issues[]` as a structured entry with `problem` and `severity`. Prose fields are silently dropped by the downstream repair pipeline; an issue you describe in prose but not in `semantic_issues[]` will never trigger a fix.
